# 크롬 확장 전환 검토 — 리뷰 요청용 정리

이 문서는 다른 세션/다른 AI가 이전 대화 맥락 없이도 바로 검토·이어갈 수 있도록 정리한 것입니다. 지금까지 나눈 대화의 결론만 담았고, 중간 논쟁 과정은 생략했습니다.

## 배경

- `campaign-strategy-os`는 서버·DB 없는 정적 SPA(React, `src/App.jsx`)로, 브라우저 localStorage에 저장하며 GitHub Pages로 공개 배포 중(https://talkspinout.github.io/campaign-strategy-os/).
- 광고·전략 기획 브리프를 "포스트잇처럼 쌓고 → 대표안으로 승격 → 논리 연결 점검 → 브리프 출력"하는 흐름의 도구. 오길비/사치&사치/JWT/레오버넷 계열 프레임워크를 재구성한 템플릿과 저자 오리지널 템플릿(`전략 전개형`)을 포함.
- 목적 재확인: **당장 수익화가 목표가 아니라 포트폴리오 강화가 목표.** "실제로 작동하는 서비스를 만들었다"는 사실 자체가 필요하며, 결제 기능은 당분간 얹지 않기로 결정함. 수익화 로드맵은 나중에 상황이 괜찮아지면 참고할 용도로만 정리해둔 것.

## 검토했다가 보류한 경로: Notion 템플릿화

같은 세션에서 이 웹 OS를 노션으로 옮기는 작업을 먼저 시도했으나(카드/브리프/체크리스트 DB, 보드+탭 구조 등 5차례 재설계), 다음 5가지가 **노션의 구조적 한계**로 확인되어 보류함(데이터 구조를 다시 짜도 해결 안 되는 문제들):

1. 뷰/탭은 같은 데이터소스 안에서만 통합 가능 — 서로 다른 DB는 탭으로 못 묶임
2. relation 속성으로는 뷰 필터가 정상 작동하지 않음(select 속성으로 우회해야 함)
3. 여러 행을 문단형 텍스트(문서)로 합쳐 보여주는 라이브 뷰가 없음
4. 데이터베이스 템플릿(+새로 만들기 드롭다운)을 API로 생성 불가 — 기존 템플릿 적용만 가능
5. 페이지 폭(전체 너비) 등 표시 설정을 API로 제어 불가

결과물(캠페인 브리프 빌더 템플릿 노션 워크스페이스, 4개 유형 예시 브리프·카드, 보드 중심 구조)은 남겨뒀지만 추가 투자는 보류 상태.

## 결정된 방향: 크롬 확장 + 구글 드라이브 동기화

- 지금 앱이 이미 서버 없는 정적 SPA라 확장 전환 비용이 낮고, 배포 채널(Chrome Web Store)이 검증돼 있고, 타깃(마케터·기획자)이 데스크톱 브라우저 사용자와 겹침.
- **결제·구독·라이선스 로직은 이번 범위에 포함하지 않음.** 전 기능 무료로 유지.

## 기술 계획 (현재 진행 대상)

1. **크롬 확장 포팅**: Manifest V3로 패키징. `localStorage` 사용 부분을 `chrome.storage.local`로 교체.
2. **구글 드라이브 연동**: 기존에 이미 있는 프로젝트 저장/불러오기 로직(`App.jsx`의 `saveProjectFile`, `importProject` — JSON 직렬화 방식)을 재사용해서, 로컬 파일 다운로드/업로드 대신(혹은 추가로) **구글 드라이브 저장/불러오기**를 옵션으로 추가. 데이터 스키마는 이미 정의돼 있어 새로 설계할 필요 없음.
3. **Google Cloud OAuth 설정**: `drive.file` 스코프(앱이 만든 파일에만 접근하는 제한된 권한) + `chrome.identity.getAuthToken` 사용. 우선 **"Testing" 상태**로 등록(테스트 사용자 최대 100명, 구글 검증 절차 불필요)해서 개발·확인 진행. 일반 공개 시점에만 verification 진행.
4. **로컬 테스트 우선**: "압축해제된 확장 프로그램 로드"로 먼저 동작 확인(저장 → 구글 드라이브 업로드 → 다른 기기에서 불러오기까지) 후, 안정화되면 Chrome Web Store 제출.

## 확인된 사실·제약 (엔지니어링 참고용)

- **Chrome Web Store**: 자체 인앱결제 API는 폐지됨. 결제가 필요해지면 Stripe/Lemon Squeezy 등 외부 연동 필요. 개발자 등록비 $5(1회). 스토어 심사는 OAuth 확장 API 검증과 별개 절차.
- **Google OAuth 검증**: `drive.file`은 "민감(sensitive)" 스코프이며 "제한(restricted)" 스코프가 아님 → 검증 자체는 무료(유료 연간 보안감사 불필요). 정식 공개 시 필요한 것: 개인정보처리방침 URL, 도메인 소유 인증, 앱 시연 영상. 심사 기간 통상 1~4주.
- **Notion API**(나중에 내보내기 기능 고려 시): 호출 자체는 무료, OAuth 연동 개발 공수만 필요.
- **Word(.docx) 내보내기**(나중에 고려 시): 외부 API 비용 없음, 순수 라이브러리로 클라이언트/서버에서 생성 가능.
- **AI 연동 기능**(나중에 고려 시): 실제 비용 발생. BYOK(사용자 자기 API 키 입력) 방식이 개발자 COGS 부담을 피하는 방법.

## 참고용 — 나중에 상황이 괜찮아지면 볼 로드맵 (현재 미실행)

1. 무료 확장 + 구글 드라이브 개인 동기화 ← **현재 여기까지만 진행**
2. 워드/노션 내보내기
3. 결제(개인 유료 티어) — Stripe/Lemon Squeezy 등. 파일 저장 자체는 서버 불필요(구글 드라이브로 충분)하지만, **결제 검증에는 최소한의 서버리스 함수**(예: Stripe 웹훅 수신용)가 필요함.
4. 팀 플랜/협업 — 이 단계부터는 진짜 공유 백엔드 필요(개인 드라이브는 팀 공유에 부적합).

## 법률·컴플라이언스 메모 (전문가 확인 필요 — 법률 자문 아님)

- **통신판매업 신고**: 무료 제공 단계에서는 불필요. 실제 유상 결제를 시작하는 시점부터 의무 발생. 정확한 면제 기준, 사업자등록 병행 여부는 결제 시작 전 세무사·법무 상담으로 확인 필요.
- **템플릿 이름**: 오길비형/사치&사치형/JWT T-Plan형/레오버넷형은 실제 광고 에이전시의 등록 상표. 상업화 시점(유료 판매 등)에는 구조 설명형 이름으로 변경 검토(예: 인식전환형/단일제안형/포지션이동형/행동변화형). 본문 서술에서 "~계열 구조에서 영감을 받음" 정도 언급은 낮은 리스크.
- **권리 매각 가정 시나리오**: 향후 이 솔루션의 권리를 매각하는 경우, 계약서에 매각 범위(소스코드/도메인/스토어 개발자 계정/상표 포함 여부)와 기존 사용자 처리 조항을 명시해야 함. 구글 드라이브 기반 저장 구조를 쓰면 서비스가 종료되더라도 사용자 데이터는 각자의 드라이브에 남는다는 장점이 있음.

## 즉시 다음 단계 (합의된 첫 착수 작업)

1. `manifest.json` 작성(Manifest V3, `chrome.identity` 권한 포함) + storage 계층을 `chrome.storage.local`로 교체
2. Google Cloud 프로젝트 생성 + OAuth 클라이언트(Chrome 확장 타입) 설정 + Drive API 활성화
3. 구글 드라이브 저장/불러오기 기능 구현(기존 JSON 직렬화 로직 재사용)
4. "압축해제된 확장 프로그램 로드"로 로컬 동작 확인
